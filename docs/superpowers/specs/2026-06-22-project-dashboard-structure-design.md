# 프로젝트 대시보드 — 구조 설계 (Structure Design Spec)

작성일: 2026-06-22
상태: 제안됨 → (사용자 승인 대기)
관련 문서: `2026-06-22-dashboard-overview.spec.md`(종합 정리)

> 이 문서는 대시보드의 **데이터 구조와 진행률 계산 규칙**만 확정한다.
> 시각화(차트·색·레이아웃) 디테일은 별도 단계(`dashboard-visualization.spec.md`)로 분리한다.

## 1. 목적

팀 작업을 세 갈래로 정리하고 진행률을 추적·갱신한다.

- **큰 과제** — 6개
- **작은 과제** — 큰 과제를 위해 별도로 해야 하는 것들(별도 그룹으로 관리)
- **개인 과제** — 내 개인 작업 (owner = 나)

진행률은 **과제 단위로만** 둔다. 그룹(목표) 전체를 합친 퍼센트는 두지 않는다.
우선순위(사용자 확정): ① **과제별 상태/담당**, ② 과제별 진행률.

## 2. 확정된 결정 (사용자 답변)

- 구조: **A안 — 3개 그룹 단일 모델** (`그룹 → 과제 → 체크리스트` 한 종류로 통일).
- 작은 과제: 큰 과제를 위해 별도로 해야 하는 것들을 모은 **별도 그룹**.
- 개인 과제: **별도 '개인' 그룹**으로 분리 (owner = 나).
- 진행률: **과제 단위로만** 둔다. 체크리스트 완료 비율 **자동 롤업** + **중요도 가중치(상/중/하)**.
- **그룹(목표) 전체 진행률 퍼센트는 두지 않는다.** 그룹은 묶음 역할만.
- 기준: 기존 모델 확장이 아닌 **이번 구조로 새로 설계**.

## 3. 데이터 모델 (3계층)

```
Group (그룹)
  id, name, kind: "team_goal" | "small" | "personal", order
  └ Task (과제)
       id, group_id, name, status, owner, memo,
       checklist[], issues[], tokens[]
       └ ChecklistItem (체크리스트 항목)
            id, text, done: bool, importance: "high" | "mid" | "low"
```

기본 그룹 3개(초기 시드):

| 그룹 | kind | 내용 |
|------|------|------|
| 큰 과제 | `team_goal` | 6개 과제 |
| 작은 과제 | `small` | 큰 과제를 위해 별도로 해야 하는 것들 N개 |
| 개인 | `personal` | 내 개인 과제 N개 (owner = 나) |

필드 정의:

- **Task.status**: `예정 | 진행 | 완료` (3단계).
- **Task.owner**: 담당자 이름 (개인 그룹은 기본값 = 나).
- **ChecklistItem.importance**: `high(상) | mid(중) | low(하)`, 기본값 = `mid`.
- **issues / tokens / memo**: 기존 오버뷰 스펙과 동일 의미 유지(문제점 목록, 토큰 기록, 자유 메모).

확장 여지: 그룹은 3개 고정이 아니라 추가 가능(예: 다음 분기 목표 → 새 `team_goal` 그룹). 모델이 한 종류이므로 추가만으로 동작.

## 4. 진행률 계산 규칙

### 4.1 중요도 가중치

| 중요도 | 가중치 |
|--------|--------|
| 상 (high) | 3 |
| 중 (mid) | 2 |
| 하 (low) | 1 |

### 4.2 과제(Task) 진행률 — 가중 평균

```
task.progress = Σ(완료된 항목의 가중치) / Σ(전체 항목의 가중치) × 100
```

예: 상(3, 완료) + 중(2, 미완) + 하(1, 완료) → (3+1) / (3+2+1) = 4/6 ≈ 67%.

예외 처리 (확정):

- `status = 완료`인 과제는 **항상 100%** (체크리스트 미완이라도 완료 우선).
- 체크리스트가 **비어 있으면** 상태 기반 폴백: `예정 = 0% / 진행 = 50% / 완료 = 100%`.

### 4.3 그룹 진행률 — 없음

그룹(큰 과제 / 작은 과제 / 개인)은 **묶음 역할만** 한다. 그룹 전체를 합친 진행률
퍼센트는 계산·표시하지 않는다. 진행률은 과제 단위로만 존재한다.

## 5. 화면 레이아웃

첫 화면은 그룹을 **따로 고르지 않고** 한 화면에 위→아래로 모두 보인다. 색/차트 등
세부 시각화는 `dashboard-visualization.spec.md`에서 다루되, 기본 골격은 다음으로 확정.

- **최상단 — 큰 과제 6개**: **3×2 사각형(카드) 그리드**로 항상 보임.
  각 카드에 과제명 + **상태/담당(우선 표시)** + 진행률 바(보조).
- **그 아래 — 작은 과제**: 목록 형태(큰 과제를 위해 별도로 해야 하는 것들).
  행마다 상태/담당 우선, 진행률 보조.
- **그 아래 — 개인 과제**: 목록 형태(owner = 나). 동일하게 상태/담당 우선.
- **펼침 편집**: 카드/행 클릭 시 체크리스트(중요도 선택 포함)·문제점·메모 편집.

표시 우선순위: 모든 위치에서 **상태/담당을 진행률보다 먼저·강하게** 노출한다.

## 6. 작업 방식 (오버뷰 스펙 규칙 계승)

- HTML 직접 수정 전 `.spec.md`에 계획 → 사용자 확인 → 구현 → 헤드리스(jsdom) 검증.
- 계획 항목 상태: `제안됨 → 승인됨 → 구현됨`.
- 외부 라이브러리 없이 SVG/CSS로 구현(오프라인 유지) — 오버뷰 스펙과 동일.
